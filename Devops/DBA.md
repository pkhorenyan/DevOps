# Репликация

Копирование данных между серверами

*Преимущество*:
Большое количество копий данных. Если головной сервер выходит из строя, любой другой сможет его заменить

*Недостаток:*
Рассинхронизация и задержки при передаче данных между серверами

Установка контейнеров mysql

Установка master:
```
docker run -d --name replication-master -e MYSQL_ROOT_PASSWORD=123456 -d mysql:8.3
```

Установка slave:
```
docker run -d --name replication-slave -e MYSQL_ROOT_PASSWORD=123456 -d
mysql:8.3
```

Для реализации взаимодействия создадим мост и сеть:

```
docker network create replication
docker network connect replication replication-master
docker network connect replication replication-slave
```

Создадим учетную запись master для сервера репликации:
```
docker exec -it replication-master mysql
```

В контейнере выполним:
```
mysql> CREATE USER 'replication'@'%';
mysql> GRANT REPLICATION SLAVE ON *.* TO 'replication'@'%';
```

Проверим права пользователя:
```
SHOW GRANTS FOR replication@'%';
```

С помощью редактора файлов вносим изменения папка etc в файл
my.cnf-> секция [mysqld] добавляем следующие параметры:

```
server_id = 1
log_bin = mysql-bin
```

При изменении конфигурации сервера требуется перезагрузка:
```
docker restart replication-master
```

После требуется зайти в контейнер и проверить состояние:
```
docker exec -it replication-master mysql
mysql> SHOW MASTER STATUS;
```

После, следует зайти в режим работы с SQL и вывести настройки
master сервера (они понадобятся при настройке slave):

```
docker exec -it replication-master mysql
mysql> SHOW MASTER STATUS;
```

**!!! Важно запомнить Значения ячеек File и Position**
Значениия File и Position меняются каждый раз.

Master готов,переходим к slave:

```
docker run --name replication-slave -e MYSQL_ROOT_PASSWORD=123456 -d
mysql:8.3
```

Открываем конфигурационный файл на Slave my.cnf.
my.cnf-> секция [mysqld] добавляем следующие параметры:
```
log_bin = mysql-bin
server_id = 2
relay-log = /var/lib/mysql/mysql-relay-bin
relay-log-index = /var/lib/mysql/mysql-relay-bin.index
read_only = 1
```

Перезагружаем slave:
```
docker restart replication-slave
```

Следующим шагом требуется прописать в базе данных на
сервер slave,кто является master и данныеFile и Position,
полученные а результате выполнения команды SHOW
MASTER STATUS; на мастере:

```
docker exec -it replication-slave mysql

mysql> CHANGE MASTER TO
MASTER_HOST='replication-master',
MASTER_USER='replication',
MASTER_LOG_FILE='mysql-bin.000001',
MASTER_LOG_POS=156;
```

Далее запускаем журнал ретрансляции и проверим статус
операций:
```
mysql> START SLAVE; (или START REPLICA;)
mysql> SHOW SLAVE STATUS\G
```

# PostgreSQL
## 1. Репликация Master-Slave (Основная модель)

В терминологии PostgreSQL это чаще называется **Primary-Replica**. Это стандартный и самый популярный метод.

- **Master (Primary):** Принимает и чтение, и запись.
    
- **Slave (Replica):** Находится в режиме `Read Only`. Она постоянно «догоняет» мастера, применяя изменения.
    

### Типы Master-Slave репликации:

1. **Физическая (Streaming Replication):** Копируются данные на уровне байтов (блоки на диске). Реплика — это точная копия мастера.
    
    - _Плюсы:_ Очень быстрая и надежная.
        
    - _Минусы:_ Нельзя реплицировать только одну таблицу (только весь сервер целиком), и версии Postgres на серверах должны быть одинаковыми.
        
2. **Логическая (Logical Replication):** Передаются конкретные изменения данных (INSERT/UPDATE/DELETE).
    
    - _Плюсы:_ Можно выбрать конкретные таблицы, можно реплицировать данные между разными версиями Postgres (например, с 12-й на 16-ю).
        

---

## 2. Модель Master-Master (Multi-Master)

«Из коробки» в стандартном дистрибутиве PostgreSQL **полноценного Multi-Master нет**.

В классическом Postgres нельзя просто поднять два сервера и писать в оба так, чтобы они мгновенно и бесконфликтно синхронизировались. Однако существуют расширения и сторонние решения:

- **BDR (Bi-Directional Replication):** Самое известное решение от компании 2ndQuadrant (теперь EDB). Позволяет записывать на несколько узлов.
    
- **Patroni + HAProxy:** Это не чистый Master-Master, а система «высокой доступности». Если мастер падает, один из слейвов мгновенно становится новым мастером.
    
- **Citrus / Postgres-XL:** Это решения для шардинга (распределения данных по разным серверам), которые часто путают с мульти-мастером.
    

---

## 3. Синхронная vs Асинхронная репликация

Это важный нюанс того, как мастер подтверждает запись:

- **Асинхронная (по умолчанию):** Мастер записал данные, ответил клиенту «ОК» и потом отправил данные на реплику. Если мастер сгорит в этот микромомент, данные могут потеряться.
    
- **Синхронная:** Мастер ждет, пока реплика подтвердит получение данных, и только потом отвечает клиенту «ОК». Это надежнее, но если реплика или сеть «тормозят», то запись на мастере тоже замедлится.


# Шардинг

Это разделение данных БД и их раздельное хранение на различных физических серверах
Существует два вида шардинга:

● вертикальный
● горизонтальный

Шардинг является наиболее предпочтительным для высоконагруженных сервисов. Но стоит
отметить, что это достаточно сложно реализовать, так как необходимо учитывать межсерверное взаимодействие.


# Резервное копирование

Виды и типы резервного копирования

● Полное резервное копирование (Full backup);
● Инкрементные бэкапы (Incremental backup);
● Дифференциальный бэкап (differential backup);
● Холодное резервирование БД;
● Горячее резервирование БД;
● Репликация.

#### Логический бэкап (Дампы)

Этот метод создает файл со списком SQL-команд для воссоздания структуры и данных базы. 

- **Утилиты:**
    - `pg_dump` — для бэкапа одной конкретной базы.
    - `pg_dumpall` — для сохранения всего кластера (всех баз, пользователей и прав).
- **Плюсы:** Можно восстановить данные на другой версии PostgreSQL или другой архитектуре ОС. Позволяет восстанавливать отдельные таблицы.
- **Минусы:** Работает медленно на больших объемах (от 100 ГБ), создает нагрузку на CPU и не позволяет восстановиться на конкретный момент времени (Point-in-Time Recovery). 

#### Физический бэкап (Слепок файлов)

Копирование самих файлов данных напрямую из файловой системы. 

- **Утилиты:**
    - `pg_basebackup` — стандартная утилита для создания базовой копии работающего кластера.
    - **LVM/Снапшоты** — моментальные снимки диска на уровне ОС или хранилища.
- **Плюсы:** Быстрое восстановление (простое копирование файлов обратно).
- **Минусы:** Бэкап привязан к версии PostgreSQL и архитектуре сервера.

#### Непрерывное архивирование и PITR

Самый надежный метод для крупных проектов. Сочетает физический бэкап и сохранение журналов транзакций (WAL-файлов). 

- **Инструменты:**
    - **pgBackRest** — считается стандартом индустрии. Поддерживает инкрементальные бэкапы, сжатие и работу с S3/Azure/GCS.
    - **WAL-G** — высокопроизводительный инструмент (преемник WAL-E), оптимизированный для работы с облачными хранилищами.
    - **Barman** — мощный менеджер бэкапов с удобным интерфейсом и возможностью удаленного управления.
    - **pg_probackup** — популярное решение с поддержкой инкрементальных копий на блочном уровне.
- **Главный плюс:** Позволяет восстановить базу на **любую секунду** в прошлом (Point-in-Time Recovery).
